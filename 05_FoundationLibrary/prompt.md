# Chapter 5 — Foundation Library: Advanced Custom Types (`QmlAdvCore`)

## Your task

You are creating the **project succession** for Chapter 5 of the book *Qt6 QML Advanced* by Daniel Gakwaya. The author writes each lecture **from working code**, so the projects must exist before the prose. Your job is to create a series of incrementally-built, self-contained Qt 6 CMake projects — one per section — that together teach the chapter's material.

You will NOT write the chapter prose. You write code only.

## What makes this chapter different

Unlike chapters 1-4, this chapter is building **a reusable library** (`QmlAdvCore`) rather than a single application. The library is the chapter's "project." Each section adds **one new advanced custom-type pattern** to the library, with a small demo app exercising it.

The library is the **final chapter of Book 1** and the **dependency that Book 2 chapters consume** (charts, QML playground, IoT dashboard). It must therefore be:

1. Genuinely reusable (clean API, no leaky abstractions, no app-specific assumptions).
2. Versioned and publishable as a standalone GitHub repository.
3. Useful even to readers who don't own Book 2 — i.e. it stands alone as a small QML utilities library.

## Chapter goal

Teach the reader the advanced `Q_PROPERTY`-family patterns that make custom QML types feel **native**, by building a library readers will actually keep using. Specifically:

- **Object properties** — child objects exposed as properties (`Theme.colors`, `Theme.spacing`).
- **List properties** — `QQmlListProperty` for QML-style child element collections.
- **Default properties** — `Q_CLASSINFO("DefaultProperty", "...")` for natural containment syntax.
- **Grouped properties** — nested property syntax (`field.validation.required: true`).
- **Attached properties** — context-aware properties added to any QML object (`Form.field: "email"`).
- **Property value sources** — types that drive property values over time, like `NumberAnimation`.
- **Non-visual services as QML types** — `QML_SINGLETON`, app-level controllers, services.
- **Library packaging** — `qt_add_qml_module` with `URI`, versioning, plugin export, consuming from external projects.

## What the reader already knows (do not re-teach)

By the time the reader reaches chapter 5, they have built:

- **Chapter 1:** `QML_ELEMENT`, `Q_PROPERTY` basics, `Q_INVOKABLE`, signals/slots both directions, context properties, declarative QML modules.
- **Chapter 2:** custom types exposed to QML (`User`, `Repository`), REST integration.
- **Chapter 3:** custom `QAbstractListModel`, custom roles, `QSortFilterProxyModel`.
- **Chapter 4:** REST-powered models, threading, caching, ETags, live updates.

Chapter 5 picks up where chapter 1 left off on **type design** and goes deep. Do not re-explain what `Q_PROPERTY` or `QML_ELEMENT` does. Show the **advanced patterns**.

### Difference 1: library + demo, not just an app

Unlike previous chapters where each section is a single app, **each section here has two parts**:

```
NN_section_name/
  QmlAdvCore/              ← the library at this section's state
    CMakeLists.txt
    src/
      theme.h / .cpp
      ...
    qml/                   ← QML helper files in the module, if any
    qmldir                 ← auto-generated by qt_add_qml_module
  demo/                    ← a tiny app that USES the library at this state
    CMakeLists.txt
    main.cpp
    Main.qml
  CMakeLists.txt           ← top-level that adds_subdirectory(QmlAdvCore) and add_subdirectory(demo)
```

The library grows monotonically across sections. The demo changes each section to exercise the *new* feature without growing into a real app — keep demos under 100 lines of QML each. The demos are throwaway; the library is the keepsake.

### Difference 2: ready for extraction to its own GitHub repo

The `QmlAdvCore/` subfolder in the **final section** (10) must be **structured so it can be lifted wholesale into a standalone repository** named `QmlAdvCore` with no edits. That means:

- Its `CMakeLists.txt` must be top-level-callable (use `if(CMAKE_SOURCE_DIR STREQUAL CMAKE_CURRENT_SOURCE_DIR)` patterns where helpful, but keep it simple — it should also work as a subdirectory of the chapter's parent project).
- A `README.md` inside `QmlAdvCore/` (final section only) explaining how to consume the library standalone via `FetchContent`, git submodule, or copy.
- A `LICENSE` file (MIT, matching the book's license) inside `QmlAdvCore/` from the first section onward.
- Versioned: `project(QmlAdvCore VERSION 0.X.0 ...)` bumping minor version each section so the reader sees `0.1.0` → `0.2.0` → … `0.10.0` across the chapter. The final section uses `1.0.0`.

### 02_theme_object_property

**Purpose:** teach **object properties** — exposing a child `QObject` as a property of a parent type.

**What to add over 01:**

- `src/theme/theme.h` / `.cpp` — singleton `Theme` exposed as `QML_SINGLETON`.
- `src/theme/themecolors.h` / `.cpp` — plain `QObject` with `Q_PROPERTY` for colors (`primary`, `surface`, `text`, etc., as `QColor`).
- `src/theme/themespacing.h` / `.cpp` — plain `QObject` with spacing scale (`xs`, `sm`, `md`, `lg`, `xl` as `int`).
- `Theme` has `Q_PROPERTY(ThemeColors* colors READ colors CONSTANT)` and `Q_PROPERTY(ThemeSpacing* spacing READ spacing CONSTANT)`.
- Demo: a window using `Theme.colors.primary` for background and `Theme.spacing.md` for margins. Toggle between two themes (`Theme.setMode("light"|"dark")`) via a button to show the children's properties emit `xxxChanged` and bindings update.
- Bump library version to `0.2.0`.

**Why this section exists:** the reader sees the pattern Qt itself uses (`font.family`, `border.width`). It also creates the `Theme` that every subsequent demo and Book 2 will consume.

### 03_form_list_property

**Purpose:** teach **list properties** via `QQmlListProperty` — letting QML declare child elements collectionally inside a custom type.

**What to add over 02:**

- `src/forms/formfield.h` / `.cpp` — `QObject` with `Q_PROPERTY` for `label`, `placeholder`, `value`, `fieldType` (enum: Text, Email, Number, Password).
- `src/forms/formgroup.h` / `.cpp` — `QObject` owning a `QQmlListProperty<FormField>` named `fields`. Must implement the four-callback `QQmlListProperty` ctor (append/count/at/clear).
- Demo: declare a `FormGroup` in QML containing several `FormField` children, then iterate them with a `Repeater` to render text inputs.
- Bump to `0.3.0`.

**Why this section exists:** the reader learns the `QQmlListProperty` callback pattern, which is non-obvious from documentation alone.

### 04_card_default_property

**Purpose:** teach **default properties** so the reader can write `Card { Text {} Button {} }` instead of `Card { contentItems: [Text {}, Button {}] }`.

**What to add over 03:**

- `src/components/card.h` / `.cpp` — `QQuickItem`-derived type with a child `QQmlListProperty<QQuickItem>` named `contentItems` and `Q_CLASSINFO("DefaultProperty", "contentItems")`.
- The card lays its children out in a column with `Theme.spacing.md` padding (uses chapter section 02's Theme).
- Demo: declare a `Card` with several children inline, no `contentItems:` keyword needed. Show that this is the same syntax Qt's `Rectangle` uses for `children`/`data`.
- Bump to `0.4.0`.

**Why this section exists:** default property is the small touch that makes a custom type feel native. The reader has used Qt types that do this every day but never built one themselves.

### 05_field_grouped_properties

**Purpose:** teach **grouped properties** — `field.validation.required: true`, `field.style.padding: 8`.

**What to add over 04:**

- Extend `FormField` (from section 03) with two grouped sub-objects:
  - `validation` (an object with `Q_PROPERTY`: `required`, `minLength`, `maxLength`, `pattern`, plus a `Q_INVOKABLE bool validate(QString)`).
  - `style` (object with `padding`, `borderColor`, `borderWidth`, `radius`).
- Both are exposed as `Q_PROPERTY(FormFieldValidation* validation READ validation CONSTANT)` etc.
- Demo: a form with two fields using inline grouped-property syntax — `FormField { validation.required: true; validation.minLength: 3 }`. Show validation result inline.
- Bump to `0.5.0`.

**Why this section exists:** grouped properties scale APIs gracefully. Once the reader sees this pattern, they'll know to use it whenever a type accumulates related options.

### 06_form_attached_properties

**Purpose:** teach **attached properties** — adding context-aware properties to objects of any type (`Form.field: "email"` like `Layout.fillWidth` or `Keys.onPressed`).

**What to add over 05:**

- `src/forms/formattached.h` / `.cpp` — implements `QML_ATTACHED` macro plus `qmlAttachedProperties` function for an attached type that any `QQuickItem` (typically `TextField`) can use.
- Attached properties: `field` (string key), `required` (bool), `pattern` (string).
- A `Form` type (or `FormController` singleton) collects all items in the QML tree that have `Form.field` set and exposes `Form.values` (a `QVariantMap`) plus a `Form.submit()` signal.
- Demo: any `TextField` with `Form.field: "email"` is automatically tracked by `Form` without subclassing — exactly how `Layout.fillWidth` works.
- Bump to `0.6.0`.

**Why this section exists:** attached properties are the most "magical-feeling" advanced pattern. Once the reader implements one, the magic becomes intuitive.

### 07_pulse_property_value_source

**Purpose:** teach **property value sources** — types that drive a target's value over time, the pattern behind `NumberAnimation`, `Behavior`, `SmoothedAnimation`.

**What to add over 06:**

- `src/effects/pulse.h` / `.cpp` — implements the property value source interface via `QQmlPropertyValueSource` and the `setTarget(const QQmlProperty&)` virtual.
- `Pulse` cycles the target between `from` and `to` over `period` ms, looping. (Pick a sane impl with a `QVariantAnimation`.)
- Usage: `Pulse on opacity { from: 0.4; to: 1.0; period: 800 }` attached to any QML item.
- Demo: a button that pulses opacity to draw attention to itself.
- Bump to `0.7.0`.

**Why this section exists:** value sources are the last remaining Q_PROPERTY-family pattern the reader hasn't met. After this section they have seen every advanced type pattern Qt uses internally.

## Constraints and don'ts

- **Do not invent new patterns the chapter doesn't promise.** The chapter teaches the eight pattern families listed above; do not add `QObject::registerMetaType` tricks, `qmlRegisterUncreatableType` workarounds, or `QML_FOREIGN` adapters unless one of the sections requires them.
- **Do not let the demos grow into apps.** Demos exist to exercise the just-added pattern; one screen, minimal QML, no router, no business logic.
- **Do not depend on chapter 4's code.** This library is intentionally independent — Book 2 will pull it as a clean dependency. (The reader's *headspace* assumes ch 1-4 knowledge; the *code* must not reach into earlier chapters' folders.)
- **Do not write the chapter prose.** Code only.
- **Do not commit to git.** The author handles commits.
- **Each section's `QmlAdvCore/` must compile and run with the prior section's demo also passing**, i.e. nothing in a later section breaks an earlier section's API. This forces additive-only design — exactly what the chapter teaches about library evolution.
